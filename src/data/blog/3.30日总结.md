# 从零手搓一个极简暗黑风 AI 聊天界面：React + CSS Modules 实践记录

最近我干了一件很朴素、也有点上头的事：  
没上 Tailwind，没上 Ant Design，没上 MUI，纯靠 React 函数组件 + CSS Modules，手搓了一个暗黑风 AI 对话界面。

目标也很明确：不是“能跑就行”的网页壳子，而是尽量往原生桌面应用的质感上靠。整体风格偏 macOS 那一挂，克制、安静、黑得很认真。

这篇文章就当一次复盘，聊聊这套界面我是怎么一步一步捏出来的。

## 这次我想做的，其实就三件事

### 1. 先别急着接大模型，前端壳子先打磨明白

这次我刻意没接真实接口，先用 Mock 数据配合 `setTimeout` 模拟 AI 回复节奏。

原因很简单：很多时候，产品体验的问题不在模型，在界面。  
输入框顺不顺手、消息流动自然不自然、页面气质稳不稳，用户一眼就能感觉出来。

所以先把“壳”做漂亮，再接“脑子”，节奏更舒服。

### 2. 设计上尽量克制，少一点“科技感渐变诈骗”

现在很多 AI 产品一打开就是蓝紫渐变铺满全屏，生怕别人不知道它和“未来”有关。

但我这次想做的是另一条路：深色、安静、克制，像个正经工具。  
不追求花里胡哨，重点放在：

- 层次感
- 呼吸感
- 微妙的透明度
- 稍微有点桌面应用味道的边界处理

说白了，就是尽量避免那种“一股网页味”。

### 3. 模块独立，不去污染原来的页面结构

整个聊天页挂在 `react-router-dom` 的 `/chat` 路由下，和原来的落地页完全分开。  
这样做的好处很直接：

- 聊天模块独立开发，不互相干扰
- 后续要单独扩展更方便
- 代码结构更清楚，不容易手滑把别的地方炸了

这点很重要。毕竟“能做出来”和“以后还能改”之间，通常隔着好几个深夜抢救现场。

---

## 整体架构：拆成五个核心组件

整个页面是一个标准的左右分栏布局，直接铺满浏览器视口，也就是 `100vh / 100vw`。  
从结构上看，核心就是五块：

- `ChatLayout`
- `Sidebar`
- `Topbar`
- `MessageList`
- `Composer`

下面一个个说。

---

## 1. `ChatLayout`：整个页面的总控中枢

这个组件可以理解成聊天页的“总导演”。  
它负责维护顶层状态，也就是那份 `messages` 数组。换句话说，它是页面唯一的数据源（Source of Truth）。

我的处理方式是典型的状态提升：

- `messages` 由 `ChatLayout` 统一维护
- 数据传给 `MessageList` 去渲染
- `Composer` 只负责收集输入并触发 `onSend`
- 真正“发消息”和“模拟 AI 回复”的逻辑，放在容器层处理

具体交互大概是这样：

1. 用户点击发送
2. 容器层先做一次乐观更新，把用户消息立即插进列表
3. 用 1.2 秒的 `setTimeout` 模拟网络延迟
4. 再塞进一条 AI 回复

这么做特别适合前期原型阶段。你不用一边写 UI，一边被接口节奏牵着跑，先把交互链路跑顺，后续接真接口会轻松很多。

---

## 2. `Sidebar`：左边栏负责稳住整个页面气质

左侧边栏我没做得很复杂，但它对界面气质影响很大。

### 视觉层级怎么做

我给它用了一个比主操作区更深一点的底色，比如 `#161616`。  
别小看这点深浅差，它会直接决定页面有没有“分区感”。

如果左右两块黑得一模一样，界面会很平，像一整块没切开的煤。

### 内容结构怎么排

边栏我分成三段：

- 顶部：应用标题 `AntiAgent` + “新建对话”按钮
- 中部：历史会话列表
- 底部：用户登录态卡片

这个结构很常规，但很稳。尤其是底部固定用户卡片后，页面会有一种“落住了”的感觉，不会漂。

### 会话列表的细节

历史会话标题做了超长截断，也就是 `ellipsis`。  
因为真实产品里，用户起标题通常非常随缘，不是“关于下周项目排期的进一步思考”，就是“啊对对对就是这个”。

所以别指望标题长度会配合你，提前截断比事后心碎靠谱。

另外滚动条我也顺手做了点哑光处理，尽量压低存在感。  
这种细节不算惊天动地，但很影响“高级感”到底是真高级，还是截图高级。

---

## 3. `Topbar`：顶部区域越克制越好

顶部固定 `56px` 高度，没有放面包屑，也没有堆很多导航元素。  
原因很简单：聊天界面的主角应该是内容区，不是头部。

所以这块只放了两类信息：

- 左边：当前会话标题
- 右边：操作按钮（分享、清空、设置、更多）

右侧按钮使用 `lucide-react` 的线框图标，颜色全部压在灰度范围，不抢戏。

背景加了一个轻微玻璃态效果：

```css
backdrop-filter: blur(12px);
```

这个处理很妙的地方在于，它不是“我来了你看我多高级”，而是你盯久了会觉得：嗯，这一层确实比纯色舒服。  
好设计很多时候就这样，不喊，但在场。

---

## 4. `MessageList`：没走文档流，而是走气泡流

这一块是我这次比较明确的取舍。

很多 AI 聊天界面喜欢做成“居中的文档流”，看起来像在和在线文档深度交流。  
这种方式没问题，但我这次更想要原生消息应用的感觉，所以用了左右气泡流。

### 用户消息

- 放在右侧，尾部对齐
- 背景用亮一点的灰色，确保与暗色环境拉开对比

### AI 消息

- 放在左侧
- 不做太重的实心底色
- 使用透明底 + 极轻白色描边：

```css
rgba(255, 255, 255, 0.08)
```

这样做的好处是：

- 不会让 AI 区域显得太重
- 能保留气泡轮廓
- 视觉上更轻、更安静

### 自动滚动

只要监听到 `messages` 更新，就调用原生滚动 API，把视图平滑滚到底部：

```ts
behavior: "smooth";
```

这是个小点，但用户感知很明显。  
如果消息更新时页面是“瞬移”到底部，会有点生硬；平滑滚动会自然很多，至少看起来不像在和一个脾气很急的 `div` 对话。

---

## 5. `Composer`：输入框才是真正的细节修罗场

如果说哪个区域最容易暴露“这产品做得糙不糙”，大概率就是输入区。  
因为用户每轮交互都要碰它，它不顺手，你前面铺的高级感都会在这里掉链子。

### 高度自适应

我用了 `textarea` 的 `scrollHeight` 动态调整输入框高度，思路很常规：

- 用户输入时先把高度重置
- 再读取 `scrollHeight`
- 然后重新设置高度
- 配一个 `max-height`，避免输入太多把页面顶飞

这类交互属于用户不一定会夸，但一定会默认你该做好的部分。

### `focus-within` 很好用

当输入框获得焦点时，我让整个外层容器一起发生一点亮度变化。  
也就是说，不只是光标在闪，整个输入区域都会轻微“醒一下”。

这种反馈非常适合深色界面，不用夸张，但一定要有回应。  
不然用户点进去之后，界面像没睡醒一样，会少很多交互上的确定感。

### 发送按钮别瞎亮

发送按钮做成纯状态驱动：

- 输入为空：禁用
- 输入全是空格：禁用
- 有有效内容：启用

这不是什么大设计，但很能体现界面是否讲道理。  
毕竟一个连空格都愿意发出去的按钮，多少有点太好说话了。

### 按压反馈

发送按钮点击时加了一个轻微位移：

```css
transform: translateY(1px);
```

就 1px，但很值。  
这种微小的“按下去”感觉，会让按钮更像按钮，而不是一张会变色的贴图。

---

## 这次最值得复盘的，其实是 CSS 细节

老实说，这套界面最后质感能不能出来，关键不在 React，反而更在 CSS。  
React 负责把结构搭起来，但“像不像成熟产品”，很多时候是透明度、边框、阴影、留白这些细节在说话。

我自己这次总结了三个很有用的点。

### 1. 别再直接上死板灰边框了

很多页面一做深色模式，边框就顺手来个 `#333`、`#2f2f2f`、`#444`。  
能用是能用，但很容易一秒进入“管理后台未完成版”的气质区间。

我这次大量使用带透明度的白色边界，比如：

```css
rgba(255, 255, 255, 0.05)
```

用在这些地方都很合适：

- 面板边缘
- `hover` 遮罩
- 消息描边
- 分隔边界

这种边界的好处是，它不是“画一根线”，而更像“给材质打一点光”。

### 2. 留白不是浪费，是气质预算

我在 `MessageList` 上留了比较大的内边距，比如：

```css
padding: 32px 48px;
```

同时把文本行高提到 `1.6`。  
看起来像在“浪费空间”，但实际效果是让界面更松弛、更稳定、不拥挤。

深色界面特别怕“密”。一密就容易压迫，一压迫就容易显廉价。  
所以别舍不得留白，你省下来的不是空间，你丢掉的是高级感。

### 3. 字体平滑真的能救命

在主入口我加了：

```css
-webkit-font-smoothing: antialiased;
```

尤其在深色背景下，这个属性会明显改善文字边缘发毛的问题。  
它不一定让你的设计起飞，但能有效避免“临门一脚毛了”。

---

## 总结

这次实践让我更确定一件事：不靠重型 UI 框架，也完全可以做出质感不错的 AI 聊天界面。

React 函数组件负责把结构组织清楚，CSS Modules 负责把样式边界收干净。  
剩下真正拉开差距的，往往是你愿不愿意花时间抠那些看起来很小的细节：

- 透明度怎么压
- 边框怎么发光
- 留白怎么呼吸
- 气泡怎么轻
- 输入框怎么回应用户

这些点单看都不大，但一层层叠起来，界面的味道就出来了。

对我来说，这次最大的价值不只是“做出了一个聊天页”，而是验证了一种更舒服的实现路径：  
先用纯前端把产品壳子打磨到位，再去接真实 LLM 接口。

这样等模型接进来时，你得到的就不是一个“能聊天的页面”，而是一个真的像产品的聊天界面。  
下一步，就该让它别再演了，接上真实接口，开始认真上班。
